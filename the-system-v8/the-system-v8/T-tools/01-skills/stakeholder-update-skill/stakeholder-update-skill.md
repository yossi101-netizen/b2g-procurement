# Stakeholder Update Skill

Write updates that busy people read in 60 seconds and know exactly where things stand.

## When to Use

- Sending weekly/monthly project status updates
- Briefing leadership or board members
- Updating clients on project progress
- Cross-functional alignment updates
- Any "here's where we are" communication

---

## Stakeholder Update Fundamentals

**What makes updates work:**
- Status is clear in the first 3 seconds (on track / at risk / blocked)
- Highlights only what changed since last update
- Risks are surfaced early (not buried at the bottom)
- Asks are specific (not "any thoughts?")

**Update length:**
- **Quick pulse:** 3-5 bullet points (for weekly standups or Slack)
- **Standard update:** Half a page (for email/doc updates)
- **Detailed briefing:** 1 page max (for board/leadership)

**The rule:** Stakeholders don't want to know everything. They want to know what matters, what changed, and what they need to do.

## The Stakeholder Update Template

### Standard Update (Most Common)

```markdown
# [Project Name] — Status Update

**Date:** [Date]
**Status:** 🟢 On Track / 🟡 At Risk / 🔴 Blocked
**Author:** [Name]

---

## TL;DR
[One sentence: Where we are and the one thing that matters most.]

---

## What Changed Since Last Update
- [Change 1: What happened and what it means]
- [Change 2: What happened and what it means]

---

## Progress

| Milestone | Status | Target Date | Notes |
|-----------|--------|-------------|-------|
| [Milestone 1] | ✅ Done | [Date] | [Note if needed] |
| [Milestone 2] | 🔄 In Progress | [Date] | [% or detail] |
| [Milestone 3] | ⬜ Not Started | [Date] | [Dependency?] |

---

## Risks & Blockers

| Risk | Impact | Mitigation | Need Help? |
|------|--------|------------|------------|
| [Risk] | High/Med/Low | [What we're doing] | [Yes: specific ask / No] |

---

## Asks (If Any)
- [Specific ask with context: "Need X from Y by Z to unblock W"]

---

## Next Steps
- [What's happening next]
- [Key dates coming up]
```

## Update Formats by Audience

### For Leadership / Board

**They care about:** Impact, risks, decisions needed, timeline.
**They don't care about:** Technical details, process updates, what you did this week.

```markdown
# [Project] — Executive Update

**Status:** 🟢/🟡/🔴
**Key Metric:** [The one number that tells the story]

**What's working:** [1 sentence]
**What's at risk:** [1 sentence, or "Nothing critical"]
**Decision needed:** [Specific, or "None — FYI only"]
**Next milestone:** [What and when]
```

### For Clients

**They care about:** Their deliverables, timeline, what they need to do.
**They don't care about:** Internal process, team dynamics, technical challenges.

```markdown
# [Project Name] — Progress Update

Hi [Name],

Quick update on where things stand:

**Completed:**
- [Deliverable] — done ✅
- [Deliverable] — done ✅

**In Progress:**
- [Deliverable] — on track for [date]

**Need from you:**
- [Specific item] by [date] to keep us on schedule

Everything is on track for [final delivery date]. Let me know if you have questions.

[Your name]
```

### For Cross-Functional Teams

**They care about:** How your work affects theirs, dependencies, timeline changes.

```markdown
# [Project] — Cross-Team Update

**Impact on your team:** [Direct, clear statement]

**What changed:**
- [Change and how it affects shared timeline]

**Dependencies:**
| We need from you | By when | Status |
|-----------------|---------|--------|
| [Thing] | [Date] | [On track / Delayed] |

| You need from us | By when | Status |
|-----------------|---------|--------|
| [Thing] | [Date] | [On track / Delayed] |
```

## Writing Rules for Stakeholder Updates

### Rule 1: Status in 3 Seconds
The very first thing they see should be: Are we on track? Use 🟢🟡🔴 traffic lights. Don't make them read a paragraph to figure out the status.

### Rule 2: What Changed > What Happened
```
Bad: "This week the team worked on the API integration and had several meetings about the design."
Good: "API integration is done (was blocked last week). Design review moved to next Monday."
```

### Rule 3: Risks Go Up, Not Down
Surface risks early and clearly. Stakeholders hate surprises. A risk shared early is manageable. A risk revealed late is a crisis.

### Rule 4: Asks Must Be Specific
```
Bad: "We could use some guidance on the timeline."
Good: "Need approval from [Name] on the revised Q2 timeline by Friday to avoid blocking dev."
```

### Rule 5: No Filler
Every sentence should add information. If removing a sentence doesn't change the update, remove it.

---

## What to Avoid

1. **"Everything is fine" when it's not** — Flag risks honestly
2. **Too much detail** — Stakeholders want altitude, not ground level
3. **No clear status** — Always lead with on track / at risk / blocked
4. **Passive voice** — "It was decided" → "We decided" / "[Name] decided"
5. **Vague asks** — "Any thoughts?" → "Need X from Y by Z"
6. **Update-only emails** — If nothing changed, don't send one. Or send: "No changes since last update. On track."

---

## Quality Checklist

Before sending:

- [ ] Status (🟢🟡🔴) is immediately visible?
- [ ] TL;DR captures the key point in one sentence?
- [ ] Only what CHANGED is highlighted (not rehashing old info)?
- [ ] Risks are surfaced with mitigation plans?
- [ ] Asks are specific (who, what, by when)?
- [ ] Adapted for the audience (leadership / client / cross-team)?
- [ ] Under 1 page?
- [ ] A busy person can scan this in 60 seconds?
